# מדריך תהליכי עבודה — מערכת סטודיו

## תהליך עבודה יומי

### 1. פתיחת סשן
כשנפתחת שיחה חדשה, ה-`session-start` hook רץ אוטומטית ומספק:
- **מצב קודם** — אם יש קובץ `production/session-state/active.md`, הוא נטען
- **בריאות פרויקט** — בדיקת node_modules, build cache, שינויים לא committed
- **פעילות אחרונה** — 5 commits אחרונים
- **זיהוי פערים** — TODOs, תיעוד חסר

### 2. עבודה על פיצ'ר
לפני כל שינוי משמעותי, השתמש ב-`/design-review`:

```
אני: רוצה להוסיף מערכת דירוג מפקד
Claude: [מריץ design-review]
  שאלה: איך מיישמים דירוג מפקד?
  אופציה א': דירוג לפי ציון סופי בלבד
  אופציה ב': דירוג מצטבר עם XP
  אופציה ג': דירוג עם כוכבים לפי קטגוריה
  המלצה: אופציה ב' (XP מצטבר)
אני: אופציה ב' נשמע טוב
Claude: [יוצר טיוטה טכנית, מחכה לאישור]
```

### 3. בדיקת איזון
אחרי שינוי בנתוני משחק, הרץ `/balance-check`:

```
אני: /balance-check
Claude: [קורא את כל קבצי ה-data וה-engine]
  [מפיק דוח מובנה עם ציונים 1-5 לכל ממד]
  [מזהה בעיות ומציע תיקונים עם ערכים ספציפיים]
```

### 4. פלייטסט
אחרי כל גרסה משמעותית, הרץ `/playtest-report`:

```
אני: /playtest-report
Claude: [מפעיל את שרת הפיתוח, מנווט דרך התרחיש]
  [מצלם מסך בנקודות מפתח]
  [מפיק דוח מובנה עם בעיות, הערות איזון, והמלצות]
```

### 5. commit
כשעושים commit, ה-`validate-commit` hook בודק:
- ✅ אין מספרים קסומים ב-engine
- ✅ אין React ב-engine
- ✅ אין enum (רק `as const`)
- ✅ אין debug code
- ✅ אין קואורדינטות hardcoded
- ✅ אין טקסט אנגלי ב-UI

### 6. סגירת סשן / דחיסת context
כשה-context מתמלא, ה-`pre-compact` hook שומר אוטומטית:
- מצב Git (branch, commit אחרון, קבצים שונו)
- אזורי עבודה פעילים (engine/ui/data/store)
- הוראות חידוש

## תהליכים ייעודיים

### הוספת סוג אירוע חדש
1. `/design-review` — הגדרת האירוע, כוחות נדרשים, הסלמה
2. הוספה ל-`src/data/event-types.ts`
3. הוספת משך טיפול ל-`src/data/treatment-durations.ts`
4. הוספת אייקון ל-`src/data/map-icons.ts`
5. `/balance-check` — וידוא שהאירוע מאוזן

### הוספת תרחיש חדש
1. `/design-review` — מבנה התרחיש, גלים, קושי
2. יצירת קובץ ב-`src/data/scenarios/`
3. `/playtest-report` — בדיקת התרחיש
4. `/balance-check` — בדיקת עקומת קושי

### שינוי מכניקת משחק
1. `/design-review` — הצגת השינוי עם אופציות
2. יישום עם Rule enforcement (engine-code / ui-code)
3. `/balance-check` — בדיקת השפעה על איזון
4. `/playtest-report` — בדיקה במשחק

## עקרונות

### הפרוטוקול השיתופי
**לעולם אל תיישם שינוי עיצובי בלי לעבור את הפרוטוקול.**

```
שאלה → אופציות → החלטה → טיוטה → אישור
```

- תמיד הצג לפחות 3 אופציות
- תמיד כלול אופציית "עשה כלום"
- תמיד חכה לאישור לפני יישום
- תמיד תעד את ההחלטה

### הפרדת אחריות
| שכבה | אחריות | כללים |
|-------|---------|-------|
| Engine (`src/engine/`) | לוגיקת סימולציה | טהור, ביצועים, אפס UI |
| Data (`src/data/`) | הגדרות סטטיות | `as const`, עברית, אפס לוגיקה |
| Store (`src/store/`) | ניהול מצב | גשר בין engine ל-UI |
| UI (`src/components/`) | תצוגה ואינטראקציה | RTL, נגישות, dark theme |

### תיעוד חי
- דוחות איזון ב-`docs/studio/balance-reports/`
- דוחות פלייטסט ב-`docs/studio/playtest-reports/`
- החלטות ב-`docs/studio/decisions/`
- כל דוח כולל תאריך והשוואה לדוח קודם
