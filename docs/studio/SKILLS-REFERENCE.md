# תיעוד Skills — מערכת סטודיו

## סקירה כללית

Skills הם פקודות `/slash` שמפעילים תהליכי עבודה מובנים. הם מוגדרים כקבצי `SKILL.md` בתיקיות ב-`.claude/skills/`.

---

## 1. `/balance-check` — סקירת איזון

**קובץ:** `.claude/skills/balance-check/SKILL.md`
**מטרה:** סקירה שיטתית של כל פרמטרי האיזון במשחק

### מתי להשתמש:
- אחרי שינוי בקבצי `src/data/` (event types, treatment durations, scenarios)
- אחרי שינוי בלוגיקת engine (escalation, scoring, weather)
- לפני playtest
- כחלק מסקירה תקופתית

### מה בודק:

| ממד | קבצים נבדקים | מה מחפש |
|-----|-------------|---------|
| לחץ זמן תגובה | `unit-types.ts`, `event-types.ts` | יחס נסיעה/הסלמה |
| כיסוי כוחות | `unit-types.ts`, `base-locations.ts` | חוסרים בכוחות |
| הוגנות הסלמה | `escalation.ts`, `events.ts` | ספירלות בלתי נשלטות |
| משך טיפול | `treatment-durations.ts` | ארוך/קצר מדי |
| השגת ניקוד | `scoring.ts` | S ריאלי, C למתחיל |
| השפעת מזג אוויר | `weather.ts` | מעניין אבל לא עונשי |
| מערכת אזרחים | `civilians.ts` | פאניקה רספונסיבית |
| עקומת קושי | `scenarios/` | עלייה חלקה |

### הפלט:
דוח מובנה בעברית ב-`docs/studio/balance-reports/[date]-balance.md`

### דוגמת שימוש:
```
אני: /balance-check
Claude: [קורא 8+ קבצים, מנתח 8 ממדים]
  📊 דוח איזון — 2026-03-23
  ציון כללי: 3.4/5 (מאוזן ברובו)

  ⚠️ בעיה: משך טיפול ב"קריסת מבנה" (240 ticks) ארוך מטיימר ההסלמה (180 ticks)
  💡 המלצה: הגדל טיימר הסלמה ל-300 ticks או קצר טיפול ל-160 ticks
```

## 2. `/playtest-report` — דוח פלייטסט

**קובץ:** `.claude/skills/playtest-report/SKILL.md`
**מטרה:** תיעוד מובנה של סשן פלייטסט

### מתי להשתמש:
- אחרי משחק בתרחיש (ידני או דרך agent-browser)
- אחרי שינוי UI משמעותי
- אחרי הוספת תרחיש חדש
- כחלק מבדיקת גרסה

### מה מתועד:

| קטגוריה | דוגמאות |
|----------|---------|
| רשמים ראשוניים | "ברור מה לעשות", "עומס מידע בהתחלה" |
| זרימת משחק | שלבי פתיחה/אמצע/שיא |
| נקודות החלטה | "לשלוח כיבוי או להמתין לפינוי" |
| תסכולים | "לחצן קשה ללחיצה", "לא הבנתי את הצבע" |
| UI/UX | קריאות, לחצנות, עומס/חוסר מידע |
| איזון | "אירוע X קשה מדי", "יחידה Y איטית" |
| ביצועים | FPS, lag, glitches |

### צילומי מסך:
נשמרים ב-`docs/dev-log/screenshots/YYYY-MM-DD-playtest-[scenario]-[desc].png`

### הפלט:
דוח מובנה ב-`docs/studio/playtest-reports/[date]-[scenario]-playtest.md`

### דוגמת שימוש:
```
אני: /playtest-report
Claude: שיחקתם תרחיש מסוים? או שאני אריץ playtest?
אני: תריץ את "רעידת אדמה בנגב"
Claude: [מפעיל dev server, מנווט לתרחיש, משחק, מצלם]
  📋 דוח פלייטסט — רעידת אדמה בנגב — 2026-03-23
  תוצאה: ניצחון — ציון 720/1000 (B)

  🔍 3 בעיות נמצאו:
  #1 [UI/בינוני] אייקון הפינוי קטן מדי במובייל
  #2 [איזון/גבוה] גל 3 מציף עם 6 אירועים בו-זמנית
  #3 [באג/נמוך] טיימר הסלמה לא מתאפס אחרי downgrade
```

## 3. `/design-review` — סקירת עיצוב

**קובץ:** `.claude/skills/design-review/SKILL.md`
**מטרה:** תהליך החלטה מובנה שמונע טעויות עיצוביות

### מתי להשתמש:
- הצעת פיצ'ר חדש
- שינוי מכניקת משחק
- עיצוב מחדש של UI
- שינוי מודל נתונים
- שינוי פילוסופיית איזון
- החלטה ארכיטקטונית

### הפרוטוקול השיתופי:

```
┌─────────┐     ┌──────────┐     ┌──────────┐     ┌────────┐     ┌─────────┐
│  שאלה   │ ──→ │ אופציות  │ ──→ │  החלטה   │ ──→ │ טיוטה  │ ──→ │ אישור   │
│ (Claude) │     │ (Claude)  │     │ (המשתמש) │     │(Claude) │     │(המשתמש) │
└─────────┘     └──────────┘     └──────────┘     └────────┘     └─────────┘
```

### כללים:
1. **תמיד 2-4 אופציות** — כולל "עשה כלום" ואופציה שאפתנית
2. **תמיד יתרונות וחסרונות** לכל אופציה
3. **תמיד מורכבות** (נמוכה/בינונית/גבוהה)
4. **לעולם לא ליישם בלי אישור**
5. **תמיד לתעד** ב-ADR

### הפלט:
ADR (Architecture Decision Record) ב-`docs/studio/decisions/ADR-[XXX]-[slug].md`

### דוגמת שימוש:
```
אני: רוצה להוסיף מערכת מוניטין לשחקן
Claude: 🔍 שאלה: איך מיישמים מוניטין?

  📋 אופציה א׳: מוניטין פשוט (ציון מצטבר)
  ✅ פשוט ליישום  ❌ לא מעניין
  מורכבות: נמוכה

  📋 אופציה ב׳: מוניטין רב-ממדי (4 קטגוריות)
  ✅ עומק  ❌ מורכב ל-UI
  מורכבות: בינונית

  📋 אופציה ג׳: מוניטין דינמי (עולה/יורד)
  ✅ מתח  ✅ אתגר  ❌ עלול לתסכל
  מורכבות: גבוהה

  💡 המלצה: אופציה ב׳ — עומק בלי תסכול

אני: אופציה ב׳
Claude: [מפיק טיוטה טכנית + ADR]
```

## יצירת Skill חדש

כדי להוסיף skill חדש:

1. צור תיקייה: `.claude/skills/[skill-name]/`
2. צור `SKILL.md` עם frontmatter:
```yaml
---
name: skill-name
description: תיאור קצר של מה ה-skill עושה
---
```
3. הגדר procedure מובנה בגוף הקובץ
4. הוסף template אם רלוונטי ב-`.claude/docs/templates/`
5. תעד ב-`docs/studio/SKILLS-REFERENCE.md`
